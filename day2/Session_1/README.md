# Session 1 — Model Serving with FastAPI

Demonstrates the full cycle: **train → save → serve → call** using the iris
dataset and FastAPI. Corresponds to sections 5.3–5.4 of
[day2_materials.md](../day2_materials.md).

## Files

```
Session_1/
├── train.py                 ← trains the model and saves artifacts to disk
├── service.py               ← FastAPI app that loads the model and serves predictions
├── iris_classifier.joblib   ← generated by train.py (gitignored)
└── label_encoder.joblib     ← generated by train.py (gitignored)
```

## Prerequisites

From the repo root, activate the project virtual environment:

```bash
source .venv/bin/activate
```

All required packages (`scikit-learn`, `fastapi`, `uvicorn`, `joblib`) are
already in `requirements.txt`.

---

## Step 1 — Train the model

Run once to produce the serialised model artifacts:

```bash
cd day2/Session_1
python train.py
```

Expected output:

```
Loaded 150 rows from iris.csv
Classes: ['Iris-setosa', 'Iris-versicolor', 'Iris-virginica']

Test accuracy: 0.9333
...
Model saved   : .../Session_1/iris_classifier.joblib
Encoder saved : .../Session_1/label_encoder.joblib
```

The pipeline saved is `StandardScaler → LogisticRegression`. The scaler and
classifier travel together as one object so the service never needs to know
about the preprocessing steps separately.

---

## Step 2 — Start the service

**From inside `day2/Session_1/`:**

```bash
uvicorn service:app --reload --port 8000
```

**From the repo root:**

```bash
uvicorn service:app --app-dir day2/Session_1 --reload --port 8000
```

`--app-dir` tells uvicorn which directory to add to `sys.path` when importing
the module — equivalent to `cd`-ing into that folder first.

`--reload` watches for code changes and restarts automatically — useful during
development. Remove it in production.

You should see:

```
INFO:     Model loaded from .../iris_classifier.joblib
INFO:     Application startup complete.
INFO:     Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
```

## Step 3 — Call the API

### Check liveness

```bash
curl http://localhost:8000/health
# → {"status":"ok"}
```

### Check readiness (model loaded?)

```bash
curl http://localhost:8000/ready
# → {"status":"ready","model_uptime_seconds":4.2}
```

### Make a prediction

Send four flower measurements as JSON:

```bash
curl -X POST http://localhost:8000/predict \
     -H "Content-Type: application/json" \
     -d '{
           "septal_length": 5.1,
           "sepal_width":   3.5,
           "petal_length":  1.4,
           "petal_width":   0.2
         }'
```

Response:

```json
{
  "predicted_class": "Iris-setosa",
  "probabilities": {
    "Iris-setosa":     0.9813,
    "Iris-versicolor": 0.0187,
    "Iris-virginica":  0.0
  },
  "model_version": "1.0.0"
}
```

### Try a virginica sample

```bash
curl -X POST http://localhost:8000/predict \
     -H "Content-Type: application/json" \
     -d '{"septal_length": 6.3, "sepal_width": 3.3, "petal_length": 6.0, "petal_width": 2.5}'
# → {"predicted_class":"Iris-virginica", ...}
```

### See input validation in action

Pydantic rejects bad inputs before they reach the model:

```bash
curl -X POST http://localhost:8000/predict \
     -H "Content-Type: application/json" \
     -d '{"septal_length": -1, "sepal_width": 3.5, "petal_length": 1.4, "petal_width": 0.2}'
# → HTTP 422  {"detail":[{"msg":"Input should be greater than 0", ...}]}
```

## What this demonstrates

| Concept | Where it appears |
|---------|-----------------|
| Model serialisation with `joblib` | `train.py` — `joblib.dump(pipeline, ...)` |
| sklearn `Pipeline` (scaler + classifier in one object) | `train.py` |
| Loading the model once at startup | `service.py` — `lifespan` context manager |
| Pydantic input validation | `service.py` — `IrisFeatures` with `Field(gt=0, le=20)` |
| Liveness vs readiness endpoints | `GET /health` vs `GET /ready` |
| Returning per-class probabilities | `service.py` — `predict_proba` |
